### 前置知识

#### 区块链的基本概念

区块链是一种去中心化的分布式账本技术，由中本聪（Satoshi Nakamoto）在 2008 年白皮书《Bitcoin: A Peer-to-Peer Electronic Cash System》中提出。

核心思想是将数据组织成"区块"，每个区块通过密码学 hash 指针链接到前一个区块，形成一条不可篡改的链条：

```
┌──────────────────────────────────────────────────────────────────────┐
│                        区块链的链式结构                                │
│                                                                      │
│  ┌──────────────┐      ┌──────────────┐      ┌──────────────┐       │
│  │   Block 0    │      │   Block 1    │      │   Block 2    │       │
│  │  (Genesis)   │      │              │      │              │       │
│  │              │      │              │      │              │       │
│  │ prev = 00..0 │      │ prev = H0 ───┼─────▶│ prev = H1 ───┼──▶... │
│  │ data = "G.." │      │ data = "A.." │      │ data = "B.." │       │
│  │ nonce = N0   │      │ nonce = N1   │      │ nonce = N2   │       │
│  │ hash = H0 ───┼─────▶│ hash = H1 ───┼─────▶│ hash = H2    │       │
│  └──────────────┘      └──────────────┘      └──────────────┘       │
│                                                                      │
│  篡改 Block 1 的数据 → H1 改变 → Block 2 的 prev ≠ H1 → 链断裂      │
└──────────────────────────────────────────────────────────────────────┘
```

区块链的关键特性：

1. **不可篡改**：修改历史区块会改变其 hash，后续所有区块的 prev_hash 都不再匹配
2. **去中心化**：无中心服务器，每个节点都有完整账本副本
3. **共识机制**：PoW/PoS 等机制决定哪个版本的链被全网接受
4. **透明可验证**：任何人都可以验证整个链的完整性

#### 什么是 SHA-256

SHA-256（Secure Hash Algorithm 256-bit）是 SHA-2 家族中最广泛使用的密码学 hash 函数。由美国国家安全局（NSA）设计，NIST 在 FIPS 180-4 中标准化。它是比特币的基石——比特币使用双重 SHA-256（SHA-256d）进行挖矿和地址生成。

SHA-256 的关键特性：

```
┌──────────────────────────────────────────────────────────────────┐
│                    SHA-256 基本性质                                 │
│                                                                  │
│  输入:  任意长度消息 M (< 2^64 位)                                 │
│  输出:  固定 256 位 (32 字节) 摘要 H(M)                            │
│                                                                  │
│  安全属性:                                                        │
│  ┌─────────────────┬───────────────────────────────────────┐     │
│  │ 抗原像 (Preimage)  │ 给定 H，找不到 M 使 H = SHA256(M)    │     │
│  │ 抗第二原像 (2nd)   │ 给定 M1，找不到 M2≠M1 使 hash 相同   │     │
│  │ 抗碰撞 (Collision) │ 找不到任意 M1≠M2 使 hash 相同        │     │
│  │ 雪崩效应           │ 翻转 M 的 1 位 → 约 128 位输出翻转   │     │
│  └─────────────────┴───────────────────────────────────────┘     │
└──────────────────────────────────────────────────────────────────┘
```

#### SHA-256 的 Merkle-Damgård 结构

SHA-256 是一种迭代 hash 函数，基于 Merkle-Damgård 构造：

```
┌──────────────────────────────────────────────────────────────────────┐
│                    Merkle-Damgård 结构 (SHA-256)                       │
│                                                                      │
│  消息 M  ──▶ 填充(pad) ──▶ 分成 512 位块 M₁, M₂, ..., Mₙ             │
│                                                                      │
│  ┌───────┐   ┌───────┐   ┌───────┐         ┌───────┐                │
│  │  IV   │   │       │   │       │         │       │                │
│  │ H(0)  │──▶│  C    │──▶│  C    │──▶ ... ─▶│  C    │──▶ H(M)       │
│  │256 位 │   │       │   │       │         │       │  256 位        │
│  └───────┘   └──┬────┘   └──┬────┘         └──┬────┘                │
│                 │           │                  │                     │
│                M₁          M₂                 Mₙ                     │
│                                                                      │
│  IV = 固定初始向量 (前 8 个质数平方根)                                 │
│  C  = 压缩函数 (64 轮, 512 位块 → 256 位状态更新)                     │
│                                                                      │
│  安全关键: 填充末尾的 64 位消息长度确保不同长度消息产生不同 hash。       │
│  这防止了长度扩展攻击 (length extension attack)。                     │
└──────────────────────────────────────────────────────────────────────┘
```

#### 工作量证明（Proof-of-Work）

PoW 是一种共识机制，要求参与者（矿工）通过计算来证明自己付出了"工作"：

```
┌──────────────────────────────────────────────────────────────────┐
│                    PoW 挖矿流程                                    │
│                                                                  │
│  矿工收集交易数据 ──▶ 构造区块头 ──▶ 尝试 nonce ──▶ 计算 hash     │
│                                                   │              │
│                                          ┌────────▼──────────┐   │
│                                          │ hash 前 4 hex      │   │
│                                          │ == "0000"?         │   │
│                                          └────┬──────────┬───┘   │
│                                               │ 是       │ 否    │
│                                               ▼          ▼       │
│                                         广播区块!   nonce++      │
│                                                     重新尝试     │
└──────────────────────────────────────────────────────────────────┘
```

PoW 的关键特性：

- **计算不可逆**：找到满足条件的 nonce 只能暴力枚举，没有捷径
- **验证容易**：任何人拿到 nonce 后只需一次 hash 计算即可验证
- **难度可调**：通过调整目标值（前导零个数）控制平均出块时间
- **经济激励**：矿工获得区块奖励和交易手续费

#### Hash 函数在区块链中的作用

Hash 函数将任意长度的输入映射为固定长度的输出：

```
┌──────────────────────────────────────────────────────────────────┐
│              Hash 函数的角色                                       │
│                                                                  │
│  ┌─────────┐    ┌──────────┐    ┌─────────┐    ┌──────────┐     │
│  │  Block 0 │───▶│ Hash(0)  │───▶│ Block 1 │───▶│ Hash(1)  │     │
│  │  数据    │    │ = H(0)   │    │  数据    │    │ = H(1)   │     │
│  └─────────┘    └──────────┘    └─────────┘    └──────────┘     │
│                                                                  │
│  H(0) 同时是:                                                     │
│    1. Block 0 的内容指纹（唯一标识 Block 0 的内容）                │
│    2. Block 1 的前驱指针（Block 1 通过 H(0) 引用 Block 0）        │
│                                                                  │
│  这种双重身份是区块链不可篡改性的数学基础。                         │
└──────────────────────────────────────────────────────────────────┘
```

真实区块链使用的 hash 函数：

| 区块链   | Hash 函数      | 输出长度 | 特性                        |
| -------- | -------------- | -------- | --------------------------- |
| Bitcoin  | SHA-256 (双重) | 256 位   | NSA 设计，广泛使用          |
| Ethereum | Keccak-256     | 256 位   | SHA-3 竞赛胜出者            |
| Litecoin | Scrypt         | 256 位   | 内存密集型，抗 ASIC         |
| 本题     | SHA-256 (简化) | 256 位   | 完整压缩 + 填充，无外部依赖 |

### 算法详解

#### SHA-256 消息填充

消息填充是 SHA-256 安全性的关键部分。FIPS 180-4 §5.1.1 规定：

```
┌──────────────────────────────────────────────────────────────────┐
│              SHA-256 填充规则                                      │
│                                                                  │
│  输入: 消息 M (N 字节), 总位数 L = N * 8                          │
│                                                                  │
│  步骤 1: 追加 0x80 (字节)                                         │
│           block[block_idx++] = 0x80;                              │
│                                                                  │
│  步骤 2: 如果 block_idx > 56:                                     │
│           用 0x00 填满到 64 字节                                   │
│           调用 sha256_transform()  ← 当前块放不下长度!             │
│           block_idx = 0                                           │
│                                                                  │
│  步骤 3: 用 0x00 填充到 block_idx == 56                           │
│           (为 8 字节长度字段预留空间)                               │
│                                                                  │
│  步骤 4: 在 block[56..63] 以大端序写入 64 位 L                    │
│           从最高字节到最低字节:                                     │
│           for (i = 7; i >= 0; i--) {                              │
│               block[56 + i] = (uint8_t)(total_bits & 0xFF);      │
│               total_bits >>= 8;                                   │
│           }                                                       │
│                                                                  │
│  步骤 5: 调用 sha256_transform() — 最后一次压缩                    │
│                                                                  │
│  示例 ("abc", 3 字节 = 24 位):                                    │
│    block[0..2] = "abc"  (0x61 0x62 0x63)                         │
│    block[3]    = 0x80                                             │
│    block[4..55] = 0x00 (52 字节)                                  │
│    block[56..63] = 0x00 0x00 0x00 0x00 0x00 0x00 0x00 0x18      │
│    → 恰好一个 64 字节块，sha256_transform 一次                     │
└──────────────────────────────────────────────────────────────────┘
```

**为什么需要填充？**

- SHA-256 的压缩函数只处理固定 512 位（64 字节）的块
- 任意长度消息需要填充到 512 位的倍数
- 长度字段防止不同长度消息产生相同 hash（否则 "abc" 和 "abc\0" 可能碰撞）

**为什么 block_idx > 56 时需要特殊处理？**

- 追加 0x80 后，当前块的剩余空间可能不足 8 字节
- 56 = 64 - 8，即预留 8 字节给长度字段
- 如果剩余空间不足，必须先填满当前块做一次压缩，再在新块中放长度

#### SHA-256 压缩函数（sha256_transform）

压缩函数是 SHA-256 的心脏。本题中此函数已完整提供（约 70 行），学员只需理解其调用时机。以下是其结构：

```
┌──────────────────────────────────────────────────────────────────┐
│              sha256_transform 工作流程                             │
│                                                                  │
│  输入: ctx->block[64] (一个 512 位消息块)                          │
│         ctx->state[8] (当前 256 位哈希状态)                        │
│                                                                  │
│  阶段 1 — 消息调度 (Message Schedule):                             │
│    W[0..15]  = 大端拆解 block[0..63] 为 16 个 uint32_t            │
│    W[16..63] = 用递推公式扩展:                                     │
│      σ0(x) = ROTR(x,7) ^ ROTR(x,18) ^ SHR(x,3)                  │
│      σ1(x) = ROTR(x,17) ^ ROTR(x,19) ^ SHR(x,10)                │
│      W[i] = W[i-16] + σ0(W[i-15]) + W[i-7] + σ1(W[i-2])         │
│                                                                  │
│  阶段 2 — 64 轮迭代 (i = 0..63):                                  │
│    每轮使用:                                                      │
│      Σ0(a) = ROTR(a,2) ^ ROTR(a,13) ^ ROTR(a,22)                │
│      Σ1(e) = ROTR(e,6) ^ ROTR(e,11) ^ ROTR(e,25)                │
│      Ch(e,f,g)  = (e & f) ^ (~e & g)       ← 选择函数            │
│      Maj(a,b,c) = (a & b) ^ (a & c) ^ (b & c) ← 多数函数         │
│      temp1 = h + Σ1(e) + Ch(e,f,g) + K[i] + W[i]                │
│      temp2 = Σ0(a) + Maj(a,b,c)                                  │
│      变量轮转: h=g, g=f, f=e, e=d+temp1, d=c, c=b, b=a, a=temp1+temp2 │
│                                                                  │
│  阶段 3 — Davies-Meyer 更新:                                      │
│    state[0..7] += (a..h)                                         │
│                                                                  │
│  输出: ctx->state[8] 被更新 (Merkle-Damgård 链式)                  │
└──────────────────────────────────────────────────────────────────┘
```

#### 区块链数据结构

每个区块包含 5 个字段：

```
┌─────────────────────────────────────────────┐
│              Block 结构体                     │
├──────────────┬──────────────────────────────┤
│ prev_hash    │ 前一个区块的 SHA-256 hash     │
│              │ (64 字符 hex 字符串 + '\0')    │
│ data         │ 交易数据（字符串指针）          │
│ nonce        │ PoW 随机数（uint64_t）         │
│ hash         │ 本区块的 SHA-256 hash          │
│              │ (64 字符 hex 字符串 + '\0')    │
│ attempts     │ 挖矿尝试次数（统计用）           │
└──────────────┴──────────────────────────────┘
```

字段关系：

- `hash = SHA256(prev_hash || data || nonce)` — hash 由其他字段计算而来
- `prev_hash` 指向前一个区块，形成链条
- `nonce` 是挖矿的"工作量证明"——找到它需要大量 SHA-256 尝试

#### 挖矿输入构造

  │ 64 hex字符 │   原始交易字符串    │ 十进制数  │

### 边界情况分析

| 场景                                   | 预期行为                                       | 原因                                                  |
| -------------------------------------- | ---------------------------------------------- | ----------------------------------------------------- |
| 创世区块 prev_hash = 64 个 '0'         | 正常挖矿，prev_hash 为全零 hex 字符串          | 创世区块没有前驱，约定 prev_hash 为 64 个 '0'         |
| data 为空字符串 ""                     | SHA-256 正常处理空输入（填充后恰好一个块）     | SHA-256 接受任意长度输入，包括零长度                  |
| nonce 很大（接近 UINT64_MAX）          | 理论上需要大量尝试，但期望次数约 65536         | 难度目标 16 位零 → 期望 2^16 次尝试                   |
| 多个 nonce 满足难度目标                | 取第一个找到的（nonce 最小的）                 | 比特币中矿工通常也取第一个满足的                      |
| 消息恰好 55 字节                       | 0x80 + 长度刚好填满 64 字节，一次 transform    | block_idx 初始 55，加 0x80 后 = 56，恰好放 8 字节长度 |
| 消息恰好 56 字节                       | 0x80 后 block_idx=57 > 56，需要跨块处理        | 当前块放不下长度，填满后先压缩，在新块放长度          |
| 消息 > 64 字节（多块）                 | sha256_update 每 64 字节自动调用一次 transform | Merkle-Damgård 结构天然支持多块                       |
| hash 恰好以 "0000" 开头但第 5 位也满足 | 只检查前 4 字符，第 5 位任意                   | 难度目标是 16 位零，不是更多                          |

### SHA-256 输出字节序详解

SHA-256 的所有整数运算使用大端序（big-endian）：

```
┌──────────────────────────────────────────────────────────────────┐
│              字节序转换位置                                        │
│                                                                  │
│  1. block[64] → W[0..15]:                                        │
│     w[i] = (block[p]<<24) | (block[p+1]<<16) |                   │
│             (block[p+2]<<8) | block[p+3]                         │
│     block[0]=0x61, block[1]=0x62, block[2]=0x63, block[3]=0x64  │
│     → w[0] = 0x61626364 (大端序)                                 │
│                                                                  │
│  2. 填充中的 64 位长度:                                           │
│     从高字节到低字节写入: for (i = 7; i >= 0; i--)                │
│     L=24 → block[56..63] = 00 00 00 00 00 00 00 18              │
│                                                                  │
│  3. state → digest[32]:                                          │
│     每个 state[i] 从 MSB 到 LSB 拆为 4 字节:                     │
│     digest[i*4+0] = state[i] >> 24   (最高字节)                  │
│     digest[i*4+1] = state[i] >> 16                               │
│     digest[i*4+2] = state[i] >> 8                                │
│     digest[i*4+3] = state[i]        (最低字节)                   │
│                                                                  │
│  4. digest → hex 字符串:                                          │
│     sprintf(hex + i*2, "%02x", digest[i])                        │
│     每个字节输出 2 位小写十六进制                                  │
│     最终 hex 字符串: digest[0]的hex, digest[1]的hex, ...          │
└──────────────────────────────────────────────────────────────────┘
```

### 常见错误

| 错误                                                                       | 后果                                    | 正确做法                                             |
| -------------------------------------------------------------------------- | --------------------------------------- | ---------------------------------------------------- |
| `sha256_final` 填充长度用错字节序（小端序）                                | 错误的 hash 输出，diff 失败             | 长度必须是 64 位大端序：`for (i = 7; i >= 0; i--)`   |
| `sha256_final` 忘记处理 `block_idx > 56` 的跨块情况                        | 长度字段位置错误，hash 永远不正确       | 追加 0x80 后检查 block_idx，>56 则填满压缩再继续     |
| `sha256_final` 输出摘要字节序错误                                          | 输出 hex 字符串不正确                   | state[i] 的 MSB 必须在 digest 的低索引位置           |
| `sha256_update` 中 `bitlen` 忘记更新                                       | 填充长度错误，hash 不正确               | 每个字节使 `bitlen += 8`（不是 += 1）                |
| `sha256_init` 忘记设置 `bitlen = 0` 或 `block_idx = 0`                     | 随机初始值导致不可预测的 hash           | 三个字段都需要初始化：state、bitlen、block_idx       |
| 创世区块 `prev_hash` 用 `memset(..., 0, ...)` 而非 `memset(..., '0', ...)` | prev_hash 是空字符串而非 64 个 '0' 字符 | 用 `memset(prev_hash, '0', 64)` 填充字符 '0' (0x30)  |
| 创世区块 `prev_hash` 忘记加 `'\0'`                                         | 字符串越界读取，未定义行为              | `chain[0].prev_hash[64] = '\0'`                      |
| `mine_block` 中 nonce 忘记递增                                             | 死循环，程序永不终止                    | 每次循环末尾 `nonce++`                               |
| `mine_block` 中 nonce 用 `uint32_t` 而非 `uint64_t`                        | nonce 可能溢出回绕（虽然本题中不会）    | 使用 `uint64_t` 与 Block 结构体定义一致              |
| 打印格式：nonce/attempts 用 `%d` 而非 `%lu`                                | 大值可能显示为负数或错误                | 使用 `%lu` 配合 `(unsigned long)` 转换               |
| 验证时忘记检查 `hash_meets_target()`                                       | 只验证了 hash 计算正确，未验证难度目标  | 验证需同时检查计算正确性和难度目标                   |
| `snprintf` 输出被截断（input 缓冲区太小）                                  | hash 输入不完整，结果错误               | input 缓冲区至少 512 字节                            |
| 验证标签写 `[OK]` 而非 `[VALID]`                                           | diff 失败                               | 严格匹配 expected_output.txt：验证通过显示 `[VALID]` |
| `#include` 缺少 `<stdint.h>`                                               | `uint32_t`、`uint64_t` 等类型未定义     | 必须 `#include <stdint.h>`                           |

### 重要知识点

1. **Merkle-Damgård 构造的安全性**：SHA-256 的安全性依赖于其压缩函数的抗碰撞性和消息填充中的长度编码。如果省略长度字段，攻击者可以利用长度扩展攻击——给定 H(M)，可以计算 H(M || padding || X) 而无需知道 M。这在实际中危害严重（如影响 HMAC 的安全性）。

2. **IV 和 K 常量的来源**：SHA-256 的初始向量来自前 8 个质数（2, 3, 5, 7, 11, 13, 17, 19）的平方根小数部分的前 32 位。64 个轮常量来自前 64 个质数（2..311）的立方根小数部分的前 32 位。这种"Nothing up my sleeve"（无隐藏设计）方式确保 NSA 没有在常量中埋后门。

3. **PoW 的概率模型**：难度目标"前 4 hex 为 0"意味着前 16 位为零。SHA-256 输出均匀分布，每个 nonce 尝试有 2^(-16) = 1/65536 的概率成功。尝试次数服从几何分布，期望值为 65536 次。但实际中位数约为 0.693 × 65536 ≈ 45426 次。

4. **Hash 指针的双重身份**：在区块链中，hash 值既是内容的"指纹"（验证数据完整性），又是结构的"指针"（链接前一个区块）。这种设计使得任何篡改都会同时破坏指纹和指针，形成双重保护。

5. **为什么 Bitcoin 使用双重 SHA-256**：Bitcoin 对区块头做 SHA-256 两次（SHA-256d）。这增加了安全性裕度，防止长度扩展攻击，并使得 ASIC 矿机的设计更复杂。在本题中我们使用单次 SHA-256 以简化实现。

6. **链验证是去中心化的基础**：在真实区块链中，每个节点都独立验证整个链。本题的验证代码模拟了这一过程——重新计算 hash、检查链接、验证难度目标。只有通过全部验证的链才被接受。

7. **attempts 统计的意义**：记录挖矿尝试次数可以帮助理解 PoW 的计算成本。在真实比特币中，全网算力约 500 EH/s（5×10^20 hash/s），每 10 分钟产生一个区块，意味着每个区块需要约 3×10^23 次 hash 尝试。

8. **uint8_t 和显式位操作的必要性**：SHA-256 的填充和输出要求精确的字节级控制。使用 `uint8_t`（而非 `char`）和显式移位操作（而非 memcpy）确保代码在所有平台上产生相同的 hash 值——这是密码学实现的黄金法则。

### 课堂讨论

1. **如果攻击者想篡改 Block 1 的数据，他需要做什么？**

   攻击者需要：(1) 修改 Block 1 的 data 字段；(2) 重新为 Block 1 挖矿（找到新 nonce 使 SHA-256 hash 满足难度目标）；(3) 因为 Block 1 的 hash 变了，Block 2 的 prev_hash 不再匹配，需要重新为 Block 2 挖矿；(4) 如果链更长，需要重新挖所有后续区块。这就是为什么区块链越长越安全——篡改的成本随确认数指数增长。在比特币中，6 个确认（约 60 分钟）被认为是安全的。

2. **为什么本题提供 sha256_transform 而让学生写 init/update/final？**

   压缩函数（transform）是 SHA-256 最复杂的部分（消息调度 + 64 轮迭代），实现它需要约 70 行精确的位操作代码。而 init/update/final 是 Merkle-Damgård 结构的"胶水代码"——它们展示了 hash 函数如何将任意长度输入转化为固定长度输出。这种分工让学生专注于理解迭代 hash 的结构，而不是被位操作细节淹没。

3. **如果难度目标改为前 8 hex 为 "00000000"（前 32 位为零），会发生什么？**

   满足条件的概率为 2^(-32) ≈ 1/43 亿。期望需要约 43 亿次 SHA-256 尝试才能挖到一个区块。在当前 CPU 上（约 10M hash/s），单个区块需要约 7 分钟。这比比特币当前难度（约 2^72）仍然低得多，但已经足够让学生感受到 PoW 的计算成本。

4. **SHA-256 消息填充中为什么用 0x80 而不是直接写 1？**

   SHA-256 按字节处理，最小追加单位是 1 字节。0x80 = 二进制 10000000，表示在消息末尾追加一个 '1' 位，然后跟若干 '0' 位。这是"位填充"（bit padding）的字节级实现。所有 SHA-2 变体（SHA-224/256/384/512）都使用相同的 0x80 填充约定。

5. **PoW 最大的缺点是什么？如何改进？**

   最大的缺点是能源消耗巨大——比特币年耗电量超过许多国家（如阿根廷、瑞典）。改进方向包括：(1) PoS (Proof-of-Stake)，以太坊已从 PoW 迁移到 PoS，能耗降低 99.9%；(2) 有用 PoW，如 Primecoin 用找素数替代无意义的 hash 计算；(3) 空间证明 (Proof-of-Space)，如 Chia 使用硬盘空间而非算力。

6. **如果两个矿工同时挖到区块怎么办？**

   这种情况称为"分叉"（fork）。两个矿工各自广播自己的区块，网络暂时分裂。其他矿工选择在其中一个区块后继续挖矿。最终，其中一个分叉变得更长，全网切换到最长链，短的分叉被丢弃（称为"孤块"，orphan block）。这就是比特币"最长链原则"。在比特币中，建议等待 6 个确认（约 60 分钟）才能认为交易不可逆。
