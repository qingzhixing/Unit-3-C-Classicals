## Lesson 50 — 可靠数据传输：停等协议（Stop-and-Wait）【标杆题】

### 课程任务

在不可靠信道上实现可靠数据传输：发送方每发一个包就等待确认（ACK），超时则重传，序号 0/1 交替区分新包和重传包。模拟丢包、损坏和 ACK 丢失三种故障，4 则消息全部成功交付后结束。

实现 `log_event()`、`sender_send()`、`sender_ack()`、`receiver_recv()`、`init()` 和 `main()` 主循环。

### 前置知识：为什么需要可靠传输？

真实网络 (IP 层) 提供的是"尽力而为"服务——数据包可能：

```
  发送方                           接收方
    │                               │
    ├──[Pkt]──→  ✗ 丢失              │   ← 路由器队列满，丢弃
    │                               │
    ├──[Pkt]──→  ⚡损坏              │   ← 电磁干扰，比特翻转
    │                               │
    ├──[Pkt]────────────────────────→│   ✓ 正常到达
    │          ←──[ACK]── ✗ 丢失     │   ← ACK 也可能丢
    │                               │
```

**停等协议的核心思想**：一次只发一个包，收到 ACK 后才发下一个。

### 协议状态机

```
  Sender FSM:                          Receiver FSM:
  ┌─────────┐                          ┌──────────┐
  │ 从应用层 │                          │ 等待包   │
  │ 取下一则 │                          │ seq_exp  │
  └────┬────┘                          └────┬─────┘
       │ seq = 0                            │
       ▼                                    │ pkt 到达
  ┌─────────┐                          ┌────▼─────┐
  │  发包   │                          │ 检查序号 │
  │ 启动定时器│                         └────┬─────┘
  └────┬────┘                         匹配  │  不匹配
       │                              ┌───┴──┐┌──┴──┐
       ├── ACK 到、序号正确 ──→ 翻 seq  │交付  ││发ACK│
       ├── ACK 丢失/超时 ──→ 重传     │发ACK ││丢弃包│
       └── ACK 序号错 ──→ 忽略       └──┬──┘└─────┘
                                       │ seq_exp 翻转
                                       ▼
                                   ┌──────────┐
                                   │ 等待包   │
                                   │ !seq_exp │
                                   └──────────┘
```

### 序号空间：为什么只要 0 和 1？

停等协议同一时刻最多只有 **1 个待确认包** 在信道中：

```
时间 →
SEND(seq=0) ────→ 等待 ACK=0 ────→ 收到 ACK → SEND(seq=1) ────→ ...
                 │
                 ├ 若超时: 重传 seq=0
                 ├ 若收到 ACK=1: 忽略！（是旧 ACK）
                 └ 若收到 ACK=0: 正确！翻 seq 为 1

只需 1 比特区分"新包"和"重传包"
```

### 数据流示例（本模拟的一次完整传输）

```
  发送方                     信道                   接收方
    │                          │                       │
  ══ msg1: HELLO — 干净交付 ════════════════════════════════
  [00] SEND seq=0 "HELLO" ─────┤                       │
    │                  OK (65%) ├──[Pkt seq=0]──→        │
    │                          │           [01] RECV seq=0 "HELLO"
    │                          │           seq_exp: 0→1
    │                          │←──[ACK seq=0]──        │
  [01] ACK seq=0 ✓  seq_snd:0→1 │                      │
    │                          │                       │
  ══ msg2: OpenCamp — 前向丢包 ═════════════════════════════
  [01] SEND seq=1 "OpenCamp" ──┤                       │
    │                LOST (25%) ✗   ← 包在信道丢失       │
  [02] LOST seq=1   retrans++   │                       │
  [02] SEND seq=1 "OpenCamp" ──┤  (重传)                │
    │                  OK       ├──[Pkt seq=1]──→        │
    │                          │           [03] RECV seq=1 "OpenCamp"
    │                          │←──[ACK seq=1]──        │
  [03] ACK seq=1 ✓  seq_snd:1→0 │                      │
    │                          │                       │
  ══ msg3: NCCL — ACK 回程丢失 → 重复包 ════════════════════
  [03] SEND seq=0 "NCCL" ──────┤                       │
    │                  OK       ├──[Pkt seq=0]──→        │
    │                          │           [04] RECV seq=0 "NCCL"
    │                          │           seq_exp: 0→1 (已交付)
    │              ACK LOST  ✗ ←──[ACK seq=0]           │   ← ACK 回程丢失
  [04] ACK-L seq=0  retrans++   │                       │
  [04] SEND seq=0 "NCCL" ──────┤  (重传, 接收方已交付过)  │
    │                  OK       ├──[Pkt seq=0]──→        │
    │                          │     [05] DUPL seq=0 (seq≠seq_exp=1)
    │                          │←──[ACK seq=0]── (重发 ACK, 不重复交付)
  [05] ACK seq=0 ✓  seq_snd:0→1 │                      │
    │                          │                       │
  ══ msg4: 2026 — 包损坏 ═══════════════════════════════════
  [05] SEND seq=1 "2026" ──────┤                       │
    │             CORRUPT (10%) ⚡──[Pkt seq=1]──→        │   ← 比特翻转
    │                          │     [06] CORR seq=1 (损坏, 丢弃不发 ACK)
  [06] SEND seq=1 "2026" ──────┤  (重传)                │
    │                  OK       ├──[Pkt seq=1]──→        │
    │                          │           [07] RECV seq=1 "2026" ✓
    │                          │←──[ACK seq=1]──        │
  [07] ACK seq=1 ✓   全部完成! (Sends:7 Retrans:3)      │
```

### 关键函数详解

#### `receiver_recv()` — 接收方核心逻辑

```
receiver_recv(pkt_seq, data, ok, send_ack, ack_seq):
  if not ok:          → 包损坏，记录 CORR，不发 ACK
  if pkt_seq == seq_exp:
    → 是新包！记录 RECV，交付，翻转 seq_exp，发 ACK
  else:
    → 是重传的旧包！记录 DUPL，重发 ACK（发送方可能上次没收到）
```

**为什么重传包也要回 ACK？** 发送方上次发的 ACK 可能在回程丢失了，重发 ACK 确保发送方能继续推进。

#### `sender_ack()` — 发送方核心逻辑

```
sender_ack(ack_seq, arrived):
  if not arrived:     → 超时，记录 TOUT，重传计数++
  if arrived:
    if ack_seq == seq_snd:
      → 序号正确！翻 seq_snd (0↔1)，推进 msg_idx
    else:
      → 序号错误，忽略（是过时的 ACK 重放）
```

### 信道模拟参数

| 参数          | 值  | 含义                              |
| ------------- | --- | --------------------------------- |
| LOSS_RATE     | 25% | 数据包在信道中丢失的概率          |
| CORRUPT_RATE  | 10% | 数据包到达但内容损坏的概率        |
| ACK_LOSS_RATE | 10% | ACK 在返回途中丢失的概率          |
| OK            | 65% | 正常到达的概率 (100% - 25% - 10%) |

### 完整逐轮追踪

本模拟精心选定随机种子，让 4 则消息**各演示一种信道故障**，难度递进：
HELLO 干净交付 → OpenCamp 前向丢包 → NCCL 的 ACK 回程丢失（引出重复包检测）→ 2026 包损坏。

| Round | 事件      | 发送方                   | 接收方          | 结果                             |
| ----- | --------- | ------------------------ | --------------- | -------------------------------- |
| 00    | SEND      | seq=0, "HELLO"           | —               | 发包                             |
| 01    | RECV      | —                        | seq=0 匹配 ✓    | 交付 HELLO，seq_exp→1            |
| 01    | ACK       | ACK seq=0 收到 ✓         | —               | seq_snd→1，推进到 OpenCamp       |
| 01    | SEND      | seq=1, "OpenCamp"        | —               | 发包                             |
| 02    | **LOST**  | 包在信道丢失！           | —               | retrans++，重传！                |
| 02    | SEND      | seq=1, "OpenCamp" (重传) | —               | 重发包                           |
| 03    | RECV      | —                        | seq=1 匹配 ✓    | 交付 OpenCamp，seq_exp→0         |
| 03    | ACK       | ACK seq=1 收到 ✓         | —               | seq_snd→0，推进到 NCCL           |
| 03    | SEND      | seq=0, "NCCL"            | —               | 发包                             |
| 04    | RECV      | —                        | seq=0 匹配 ✓    | 交付 NCCL，seq_exp→1             |
| 04    | **ACK-L** | ACK 回程丢失！           | —               | retrans++，发送方等不到 ACK 重传 |
| 04    | SEND      | seq=0, "NCCL" (重传)     | —               | 重发（接收方其实已交付过）       |
| 05    | **DUPL**  | —                        | seq=0 ≠ 期望 1  | 识别为重复包！重发 ACK，不重复交付 |
| 05    | ACK       | ACK seq=0 收到 ✓         | —               | seq_snd→1，推进到 2026           |
| 05    | SEND      | seq=1, "2026"            | —               | 发包                             |
| 06    | **CORR**  | —                        | 包损坏 ✗        | retrans++，丢弃不发 ACK → 重传   |
| 06    | SEND      | seq=1, "2026" (重传)     | —               | 重发包                           |
| 07    | RECV      | —                        | seq=1 匹配 ✓    | 交付 2026，完成！                |
| 07    | ACK       | ACK seq=1 收到 ✓         | —               | 全部完成（Sends:7 Retrans:3）    |

### 常见错误与陷阱

| 错误                   | 为什么错                | 正确做法                             |
| ---------------------- | ----------------------- | ------------------------------------ |
| 接收方收到所有包都交付 | 重传的旧包会重复交付    | 用 seq_exp 过滤，只交付序号匹配的包  |
| ACK 不检查序号         | 旧 ACK 可能被当成新 ACK | `if (ack_seq == seq_snd)` 检查       |
| 超时后不重传           | 包丢失后永远等不到 ACK  | `arrived==0` 时记录 TOUT 并 continue |
| CORRUPT 时还发 ACK     | 损坏的包不应确认        | `ok==0` 时 `*send_ack=0` → return    |
| sender_ack 中忘翻 seq  | 序号永远是 0            | `seq_snd = 1 - seq_snd`              |

### 重要知识点

- **停等协议**: 一次一发，等 ACK 再发下一个，简单但信道利用率低
- **序号 0/1 交替**: 1 比特序号区分新包和重传包（停等协议的特性）
- **超时重传**: 在不可靠信道上保证可靠性的核心机制
- **ACK 确认**: 接收方显式告知发送方"收到了"
- **幂等交付**: 接收方用序号保证每条消息只交付一次
- **不可靠信道模拟**: 用内置确定性 PRNG（xorshift32，固定种子）模拟三种信道事件（丢包/损坏/正常），保证跨平台可复现

### 课堂讨论

1. **停等协议的信道利用率有多低？** 设 RTT=30ms，包大小=1KB，带宽=1Gbps。发送 1KB 只需 8μs，但等待 ACK 要 30ms。利用率 = 8μs/30ms ≈ 0.027%！这就是为什么需要滑动窗口。

2. **如果 ACK 永远不来怎么办？** 定时器超时→重传。设置最大重传次数，超过则报错（本练习用 MAX_SENDS=30 做安全上限）。

3. **为什么序号不需要更多比特？** 停等协议中窗口大小 = 1，同一时刻只有一个未确认包，1 比特足够。Go-Back-N 或选择性重传则需要更大的序号空间。

4. **真实 TCP 的可靠传输和本模拟有何不同？** TCP 使用滑动窗口（非停等）、累计 ACK、快速重传、流量控制和拥塞控制。本题是最简版。

5. **如果发送方和接收方的初始 seq 不同会怎样？** 双方需要握手协商（类似 TCP 三次握手），否则第一个包会被接收方当成"序号不匹配"丢弃。
