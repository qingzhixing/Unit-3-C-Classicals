## Lesson 65 — 标记 - 清除垃圾回收器 (Mark-Sweep GC)

### 课程任务

实现一个基于标记 - 清除 (Mark-Sweep) 算法的微型垃圾回收器。在包含 16 个对象的固定堆上，用 DFS 递归标记从根集合可达的对象，然后清除（回收）所有未标记对象。

```c
/* 需要实现的 5 个部分 */
void mark_recursive(int idx);   /* 从 idx 出发 DFS 标记所有可达对象 */
int  sweep(void);               /* 回收未标记对象，返回回收数量      */
void print_objects(const char *label);  /* 打印堆中所有对象状态      */
void gc_collect(void);          /* 协调标记 + 清除两个阶段            */
int  main(void);                /* 驱动：初始化 → 打印 → GC → 打印 */
```

验证运行：

```
./mark_sweep
```

输出将展示 GC 前所有对象状态、标记阶段从每个根出发的遍历、清除阶段回收的对象列表、GC 汇总统计、以及 GC 后最终对象状态。

### 前置知识

#### 为什么需要垃圾回收？

C 语言中程序员手动 `malloc`/`free`，容易出错：

| 错误类型     | 示例                    | 后果            |
| ------------ | ----------------------- | --------------- |
| 内存泄漏     | `malloc` 后忘记 `free`  | 内存逐渐耗尽    |
| 悬空指针     | `free` 后继续使用指针   | 未定义行为/崩溃 |
| 双重释放     | 同一指针 `free` 两次    | 堆损坏/崩溃     |
| 释放错误地址 | `free` 栈变量或偏移指针 | 堆损坏/崩溃     |

垃圾回收 (Garbage Collection, GC) 让系统自动判断对象是否不再需要并回收，
消除了上述所有手动内存管理错误。

#### GC 简史

```
1959  John McCarthy  发明 GC (Lisp 语言)
1960  Collins        引用计数 (Reference Counting)
1970  Dijkstra 等    三色标记理论
1975  Steele         复制 GC (Cheney 算法)
1984  Ungar          分代 GC (Generational GC)
2000+ Java/Go/Python 生产级并发/并行 GC
```

#### 根集合 (Root Set) 概念

GC 的起点是**根集合**——程序可以直接访问、无需经过指针间接引用的对象集合：

```
┌─────────────────────────────────────────────────────────┐
│                    程序执行状态                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ 全局变量  │  │ 栈局部变量 │  │ CPU寄存器 │              │
│  │ glob_ptr │  │ loc_ptr  │  │   R1     │              │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘              │
│       │             │             │                     │
│       ▼             ▼             ▼                     │
│   根集合 = {这些变量指向的所有对象}                       │
└─────────────────────────────────────────────────────────┘
```

本题根集合简化为：`roots[] = {OBJ0, OBJ3, OBJ5}`

#### 可达性 (Reachability)

从根集合出发，通过指针引用可以到达的所有对象称为**可达对象**:

```
根 → A → B → C    (A、B、C 都是可达的 ✓)
根 → D            (D 是可达的 ✓)
   E              (E 没有路径从根到达 → 垃圾 ✗)
```

**核心定理**: 可达对象 = 将来可能被程序访问 → 必须保留；不可达对象 = 垃圾 → 可以安全回收。

### 算法深度剖析

#### 标记 - 清除两阶段

标记 - 清除 GC 分为两个阶段：

```
阶段 1: 标记 (Mark)
  ┌─────────────────────────────────────────┐
  │ 从根集合出发，遍历所有引用链，           │
  │ 为每个可达对象打上 marked 标记。          │
  │                                         │
  │ 根 → 标记根 → 遍历根的引用 → 递归标记    │
  └─────────────────────────────────────────┘
                    ↓
阶段 2: 清除 (Sweep)
  ┌─────────────────────────────────────────┐
  │ 线性扫描整个堆，回收所有未标记的对象。    │
  │                                         │
  │ for 每个对象:                           │
  │   if not marked → 回收                  │
  │   if marked → 保留 (清除标记以备下次GC)  │
  └─────────────────────────────────────────┘
```

#### 三色标记示例

Dijkstra 的三色抽象将标记过程描述为颜色转换。以下是本题从 OBJ5 出发的详细追踪：

```
初始状态 (所有白色):
  OBJ5=○  OBJ6=○  OBJ7=○

Step 1: 从根 OBJ5 出发 → OBJ5 变灰 (正在处理)
  OBJ5=◉  OBJ6=○  OBJ7=○

Step 2: 处理 OBJ5 的子节点 OBJ6 → OBJ6 变灰, OBJ5 变黑
  OBJ5=●  OBJ6=◉  OBJ7=○

Step 3: 处理 OBJ6 的子节点 OBJ7 → OBJ7 变灰, OBJ6 变黑
  OBJ5=●  OBJ6=●  OBJ7=◉

Step 4: 处理 OBJ7 的子节点 OBJ5 → OBJ5 已是黑色, 跳过!
  (这是处理循环引用的关键!)
  OBJ7 变黑
  OBJ5=●  OBJ6=●  OBJ7=●

最终: 三个对象全部标记为黑色 (可达)
```

本题简化：使用 `marked` 位 (0/1) 实现二色标记，递归调用栈隐式替代灰色队列。

### 边界情况分析

| 场景                 | 行为                            | 本题是否涉及  |
| -------------------- | ------------------------------- | ------------- |
| 循环引用             | marked 检查防止无限递归         | ✓ (OBJ5↔OBJ7) |
| 孤立对象             | 不可达，被 sweep 回收           | ✓ (OBJ8)      |
| 空堆 (无对象)        | 遍历为空，collected=0           | 边界情况      |
| 全可达 (无垃圾)      | sweep 无回收，collected=0       | 边界情况      |
| 全垃圾 (根集合空)    | 所有对象被回收                  | 边界情况      |
| 单个对象自引用       | marked 检查防止循环             | 未涉及        |
| 多根重复引用同一对象 | marked 检查防止重复标记         | 未涉及        |
| 引用越界索引         | 边界检查 (idx<0 or >=HEAP_SIZE) | 防御性检查    |

### 常见错误

| 错误                                       | 后果                       | 正确做法                                       |
| ------------------------------------------ | -------------------------- | ---------------------------------------------- |
| 忘记 marked 检查导致循环引用无限递归       | 栈溢出 (Stack Overflow)    | `if (heap[idx].marked) return;` 作为第一道检查 |
| sweep 中打印格式 `%d` 而非 `%2d`           | 输出列不对齐，diff 不通过  | 使用 `%2d` 保持宽度为 2 的右对齐格式           |
| OBJ 编号格式 `%d` 而非 `%-2d`              | 输出列不对齐               | 使用 `%-2d` 左对齐宽度 2                       |
| 忘记在 sweep 中将 ref_count 清零           | 回收对象显示非零 ref_count | `heap[i].ref_count = 0`                        |
| sweep 中忘记重置 refs[0]/refs[1]           | 回收对象仍显示旧引用       | `heap[i].refs[0] = heap[i].refs[1] = -1`       |
| gc_collect 中 alive 用 HEAP_SIZE-collected | 语义错误 (恰好本题相等)    | 遍历统计 `marked==1` 的对象                    |
| main 中忘记调用 init_heap()                | 堆全为零，输出无意义       | 第一步调用 `init_heap()`                       |
| 循环引用的 marked 检查放在标记之后         | 仍然无限递归               | marked 检查必须在标记之前                      |
| 递归调用 mark_recursive 前不检查边界       | 数组越界访问               | `if (idx < 0                                   |     | idx >= HEAP_SIZE) return;` |
| 打印 label 忘记换行                        | 输出粘连在一起             | `printf("%s\n", label);`                       |

### 重要知识点

1. **标记 - 清除是最基础的追踪式 GC 算法**。几乎所有现代 GC 都是它的变体或组合。理解它才能理解分代 GC、增量 GC、并发 GC 等高级话题。

2. **marked 位的作用不止于标记存活**。在处理循环引用时，marked 位充当 "visited 集合"，防止 DFS 无限递归。这是图的深度优先遍历中 visited 数组的经典应用。

3. **碎片是标记 - 清除的根本缺陷**。因为对象不移动，回收后空闲空间散布在存活对象之间。下次分配大对象时可能"有空间但无法分配"。标记 - 整理和复制 GC 通过移动对象解决此问题。

4. **根集合的准确性决定 GC 的正确性**。如果漏掉一个根，该根引用的所有对象都会被错误回收（程序崩溃）。如果多包含一个根，该根引用的垃圾对象会保留（内存泄漏）。

5. **递归 DFS vs 迭代 DFS**。本题使用递归实现，简洁但受限于调用栈深度。生产级 GC 使用显式栈 (gray stack) 的迭代实现，避免栈溢出并支持增量标记。

6. **标记位可以压缩存储**。每个对象只需 1 bit 的标记位。可以放在对象头部 (header) 中，或使用独立的 bitmap。本题用 int 是为了代码清晰。

7. **stop-the-world 暂停**。标记 - 清除 GC 在执行时需要暂停程序（mutator），因为程序可能在标记期间修改对象图。现代 GC 使用写屏障 (write barrier) 实现并发标记。

8. **GC 与 C 语言**。C 标准没有 GC，但 Boehm GC 库为 C/C++ 提供了保守式 GC（conservative GC），它将栈和寄存器中"看起来像指针"的值都当作根。本题是精确式 GC（precise GC）的简化版。

### 课堂讨论

1. **如果去掉 marked 检查，程序会发生什么？**

   在 OBJ5→OBJ6→OBJ7→OBJ5 的循环中，`mark_recursive` 会无限递归调用自己，每次调用消耗栈空间。最终栈溢出，程序收到 SIGSEGV 信号崩溃。这是一个经典的"图遍历忘记 visited 集合"错误。

2. **为什么标记 - 清除不移动对象，而复制 GC 要移动对象？**

   标记 - 清除的设计目标是简单和不改变对象地址（C 风格指针可以保持不变）。但代价是碎片。复制 GC 通过将所有存活对象复制到新空间并更新所有引用，消除了碎片，但需要额外空间和精确的指针更新机制。这是"简单但低效"vs"复杂但高效"的经典权衡。

3. **三色标记中的灰色集合在本实现中是如何体现的？**

   本实现没有显式的灰色集合。递归调用栈隐式地充当了灰色集合——栈上的每个帧代表一个"已标记但子节点尚未完全处理"的对象。当递归返回时，该对象从灰色变为黑色。这是 DFS 实现的三色标记。

4. **如果根集合包含 OBJ8（孤立对象），GC 结果会怎样？**

   OBJ8 会被标记为可达（marked=1），从而不会被 sweep 回收。但它仍然没有任何入边或出边。这会变成一个"僵尸对象"——存活但程序无法访问（除了通过根集合）。这说明根集合的精确性至关重要。

5. **如何将本实现改为迭代版本以避免栈溢出？**

   使用显式栈 (gray stack): 一个整数数组作为栈，将根推入栈，循环弹出栈顶，标记对象，将其未标记的子节点推入栈。这样可以处理任意深度的对象图，但代码更复杂。

6. **碎片问题有多严重？举一个具体例子。**

   假设堆有 100 字节，分配了 10 个 8 字节对象后回收其中 5 个交替的对象。存活对象：A(0-7), C(16-23), E(32-39), G(48-55), I(64-71)。空闲区间：8-15, 24-31, 40-47, 56-63, 72-99。总空闲 = 5×8 + 28 = 68 字节，但最大连续空闲块只有 28 字节。如果要分配 30 字节对象，即使总空闲 68 字节也会失败。
